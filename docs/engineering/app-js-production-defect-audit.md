# App.js Production Defect Audit

**Date:** 2026-07-21
**Baseline:** `eaf5295` (`feat/inspector-case-notes`)
**Scope:** Confirmed production reliability defects in `src/app.js` and directly related modules.
**Disposition:** Stabilize the current architecture before any modularization.

## Executive summary

The current runtime is not safe to classify as fully stabilized. Eight production defects were
confirmed: three High and five Medium. The High findings cross data or workspace boundaries and
block modularization preparation until fixed:

1. undo/redo history survives user and workspace replacement;
2. the combined legacy local-storage migration can destroy packs and cases; and
3. per-tab workspace epochs can reject a valid change from a newer tab.

The defects below are behavioral findings with a concrete source mechanism and trigger path.
File size, closure density, style, and possible future extraction boundaries are not classified as
defects.

## Confirmed defects

### DEF-001 — Undo history crosses user/workspace boundaries

- **Severity:** High
- **Affected files:** `src/core/state-store.js`, `src/app.js`
- **Trigger path:** Change cases, packs, folders, or preferences in scope A; switch user or
  workspace to scope B; invoke Undo or Redo.
- **User impact:** State from A can appear in B and then be persisted under B, crossing a user or
  workspace data boundary.
- **Evidence:** `StateStore.replace(..., { skipHistory: true })` replaces state without rebasing the
  module-level history stack. All scoped loads and empty-state resets use that path. `undo()` and
  `redo()` merge the retained snapshot into the current state. A direct runtime reproduction
  restored workspace A's case library after workspace B had been loaded.
- **Recommended minimal fix:** Add a StateStore history-reset boundary and request it only for
  identity/workspace state replacement. Preserve ordinary undo/redo within one scope.
- **Status:** Confirmed; assigned to Phase 1.

### DEF-002 — Legacy storage migration can destroy packs and cases

- **Severity:** High
- **Affected files:** `src/core/storage.js`, `src/app.js`
- **Trigger path:** Start the current application with a pre-split `truckPacker3d:v1` payload that
  contains preferences, cases, packs, folders, and the current pack.
- **User impact:** Locally stored customer packs and cases can be permanently replaced by demo
  content.
- **Evidence:** The migration copies the combined payload to a user key and immediately removes
  the source. `load()` reads cases/packs only from the workspace payload, so it ignores the copied
  arrays. Startup then seeds and saves demo data. Both module-level and browser reproductions
  showed the legacy arrays disappearing while preferences survived.
- **Recommended minimal fix:** Treat combined payload data as a migration candidate, keep it
  readable until an authenticated workspace exists, write and verify the workspace/user split,
  and remove the source only after verification.
- **Status:** Confirmed; assigned to Phase 1.

### DEF-003 — Pending autosave is lost during scope transitions

- **Severity:** Medium
- **Affected files:** `src/core/browser.js`, `src/core/storage.js`, `src/app.js`
- **Trigger path:** Make a persisted edit and switch workspace/user or log out within the 250 ms
  autosave debounce window.
- **User impact:** The last local edit can be silently lost.
- **Evidence:** The debounce primitive exposes no flush/cancel operation. `saveNow()` resolves both
  state and scope only when the delayed callback runs. Scope transitions replace state and scope
  without first completing the pending save. A direct reproduction left scope A at its previous
  value after an A edit followed immediately by a switch to B.
- **Recommended minimal fix:** Add flush/cancel controls to the existing debounce function, expose
  a focused Storage flush, and invoke it before every destructive identity/workspace replacement.
- **Status:** Confirmed; assigned to Phase 1.

### DEF-004 — Workspace epoch ordering rejects valid changes from fresh tabs

- **Severity:** High
- **Affected files:** `src/app.js`
- **Trigger path:** An established tab reaches a higher local org epoch; the ephemeral sync key
  expires; a fresh tab starts at epoch zero and performs a valid workspace switch.
- **User impact:** The established tab can remain on stale workspace, billing, members, editor, and
  storage state.
- **Evidence:** Epochs are generated by a per-tab counter, but incoming epochs are compared as if
  globally ordered. A fresh tab's epoch 1 is rejected by a tab that has already applied epoch 8.
  The canonical storage event also cancels the legacy refresh fallback before confirming that the
  canonical payload was accepted.
- **Recommended minimal fix:** Preserve the sync model but derive a monotonic logical version from
  epoch and timestamp, with a deterministic tab tie-break. Cancel the fallback only after an incoming payload is
  accepted.
- **Status:** Confirmed; assigned to Phase 2.

### DEF-005 — Packs/Cases mutations bypass the operation lifecycle

- **Severity:** Medium
- **Affected files:** `src/app.js`, `src/screens/packs-screen.js`,
  `src/screens/cases-screen.js`, `src/screens/editor-screen.js`,
  `src/ui/overlays/import-pack-dialog.js`, `src/ui/overlays/import-cases-dialog.js`,
  `src/ui/overlays/case-modal.js`, `src/services/autopack-engine.js`
- **Trigger path:** Start AutoPack or another protected editor mutation, navigate to Packs or
  Cases, and perform a conflicting mutation such as duplicate, delete, edit, import, or clear
  preview.
- **User impact:** A user action can race with operation completion, be silently overwritten, or
  cause the running operation to fail against removed/replaced data.
- **Evidence:** `OperationLifecycle` is not supplied to Packs/Cases or their import dialogs.
  Navigation remains available. AutoPack schedules its final preview after the protected operation
  releases, while `clearPackPreview()` and screen mutations have no busy guard. Focused follow-up
  inspection also confirmed uncovered editor commit edges for visibility, category, notes, manual
  position, and the shared New Case modal; these bypass the editor's existing busy helper.
- **Recommended minimal fix:** Inject the existing lifecycle into Packs/Cases and dialogs, guard
  every commit edge, reuse the editor's existing helper for its uncovered paths, and guard preview
  clearing. Keep viewing and navigation available.
- **Status:** Confirmed; assigned to Phase 3.

### DEF-006 — Invite token is cleared after retryable failure

- **Severity:** Medium
- **Affected files:** `src/app.js`, `src/data/services/billing.service.js`
- **Trigger path:** Accept a valid invite while the network or invite Edge Function has a transient
  failure.
- **User impact:** The user loses the pending token and must recover the original URL or request a
  replacement invite.
- **Evidence:** `pendingInviteToken` and its session-storage copy are removed immediately after
  `acceptOrgInvite()` returns, before checking `result.ok`. The service represents network/server
  failures as `{ ok: false }`; the catch path also removes the token.
- **Recommended minimal fix:** Centralize token clearing and invoke it only after success or a
  confirmed terminal response (expired, revoked/no-longer-valid, not found, or wrong email).
  Preserve the token for retryable failures.
- **Status:** Confirmed; assigned to Phase 4.

### DEF-007 — Billing dead-tab retry occurs before lock expiry

- **Severity:** Medium
- **Affected files:** `src/app.js`
- **Trigger path:** One tab obtains the billing lock and crashes; another tab attempts refresh while
  the 20-second lock remains.
- **User impact:** Entitlement and Settings Billing can remain stale until an unrelated focus,
  navigation, or connectivity event triggers another refresh.
- **Evidence:** A blocked tab retries exactly once after 1.2 seconds. That retry is still inside the
  20-second TTL, and retry-tagged requests cannot schedule another attempt. A crashed owner emits
  no result or removal event when the lock expires.
- **Recommended minimal fix:** Derive the one bounded retry delay from the remaining lock TTL, with
  a small post-expiry margin. Existing shared-result listeners remain authoritative.
- **Status:** Confirmed; assigned to Phase 2.

### DEF-008 — Logout retains a timer-driven reload path

- **Severity:** Medium
- **Affected files:** `src/app.js`, `src/core/supabase-client.js`,
  `tests/audit/security-and-invariants.spec.mjs`
- **Trigger path:** Supabase sign-out or its normal auth callback is slow/hung after local
  finalization.
- **User impact:** Logout can forcibly reload the application rather than completing through the
  canonical signed-out lifecycle.
- **Evidence:** The auth wrapper locally finalizes after an 800 ms wait. `app.js` then schedules a
  2.2-second fallback that reloads unless both the body auth marker and overlay are settled. The
  existing source test searches only a limited range after the direct sign-out call and misses the
  indirection through `scheduleLogoutFallbackReload()`.
- **Recommended minimal fix:** Remove the timer-driven reload and make the existing custom
  signed-out event execute deterministic app cleanup and signed-out rendering.
- **Status:** Confirmed; assigned to Phase 4.

## Risks requiring future validation

- PDF export entitlement readiness during hidden-tab or slow workspace refresh needs authenticated
  browser validation; no defect is asserted here.
- The profile-status fail-open policy for non-401 service errors needs a product/security decision
  and live failure injection.
- Recovery after a transient top-level `init()` rejection requires controlled failure injection.
- Authenticated owner/member, same-tab/cross-tab, billing, and invite browser matrices remain part
  of final stabilization QA, not this surgical implementation pass.

## False positives reviewed

- `src/app.js` size, globals, closures, and warning count are not runtime defects.
- The historical cross-user billing contamination finding is covered by current identity isolation,
  epoch checks, org/user guards, and sanitized shared snapshots; current regression tests pass.
- Owner-only billing actions are enforced by the backend; callable frontend functions alone are not
  an authorization defect.
- Shared billing snapshots remove user-specific billing-management authority; no privilege leak was
  demonstrated.
- Existing AutoPack, Unpack, Truck Change, drag, nudge, rotate, delete, and keyboard paths use the
  shared lifecycle. DEF-005 includes the confirmed Packs/Cases gap and the small set of uncovered
  editor commit edges listed above; it does not justify a second lock or library redesign.
- A local missing `favicon.ico`, source-file size, and existing lint warnings are not production
  reliability defects.
